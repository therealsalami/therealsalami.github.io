Modern Recurrent Neural Networks

Part 1: Basic RNNs

In the following section, I adjust the following Hyperparameters in order to determine their effect on the model's performance:


- Learning rate

- Number of Hidden Units

- Number of time steps in minibatch

- Number of Epochs

Part (a): Learning rate

Learning rate controls how much a model's internal weights are updated during each iteration. We can think of this as "how big of a step are we taking?"
The values we will explore for learning rate: [.1, .5, 1, 2, 5]

It seems that the learning rate of 2 seemed to do the best out of all of the learning rates that I tested. The general trend shows that with a higher learning rate, the perplexity drops off more sharply. With a very small learning rate, the training was more stable, but it resulted in longer time needed to reach the end On the topic of LR = 5... holy gradient explosion. Even with the gradient clipping, with a learning rate that high, jumps were exponential. The perplexity exploded several times, and early on too.

Number of hidden Units

The number of hidden Units allows for control over the complexity (dimensionality) that our model can handle.
The values we will explore for Hidden Units: [8, 16, 32, 64]

The general behavior of the hidden units is about the same relative to Learning rate, so there are not large, noticeable changes in the beginning of training. What is worth noting is that as the number of hidden units increases, the perplexity is able to drop further in the later stages of training. Interestingly, the gap between training perplexity and validation perplexity is larger when there are more hidden units. This leads me to believe that the model is becomming overfitted to the training set

Number of time steps in a Minibatch

Number of time steps in a minibatch controls the temporal length of each sequence sample provided to the model during a single training iteration.
The values we will explore for time steps: [8, 16, 32, 64]

In this I do not see a major difference.

Number of Epochs

An Epoch is one instance of passing the entirety of the dataset through the model. "how many repetitions are we going to do?"
The values we will explore for Epochs: [10, 50, 100, 250]

With more epochs, the model is able to reach a lower perplexity. However, after a certain point, it becomes saturated additional gains are negligible. In particular, any improvement beyond 100 epochs is negligible, as the perplexity decreases by 2 over 150 additional epochs. The validation set also reaches a plateau and no further improvement on generalization is made

Taking the Best from the previous

Just for fun, lets take the parameters that resulted in the lowest perplexity in each category, and put them in one. Number of Epochs: 250 Number of hidden Units:64 number of time steps in minibatch: 64 learning rate:2

Woah! Interesting result! The training set reaches a very low perplexity very early. (Perplexity of 5 is reached at just 50 epochs). The perplexity continues to drop to an even smaller value as the epochs grow. But beyond 50 epochs, the perplexity actually begins to grow on the validation set. I believe that there is severe overfitting to the training set and that genearlization gets worse.

3 Best Models

These three models have the best performance overall on the training set, with the "just for fun" model reaching the lowest perplexity. The validation set, however has a perplexity that is growing, meaning that I think overfitting is at play here. In terms of the base model with 250 epochs, it also has a low perplexity score which is due to the amount of epochs that the model undergoes. It has more time with the data so it can pull out more features and find smaller changes and micro adjustments. For the model with the 64 hidden layers, it reaches a low perplexity score due to the complex shapes that it can create. Since it has more layers, the descision boundary is more complex and able to maneuver more.

Part (b): Effect of Gradient Clipping and ReLU

What is the effect of gradient clipping?

Hmm... It seems to be a tad but more unstable, but not super noticeable. Let's try again with a higher learning rate (LR = 5)

Gradient clipping is definitly necessary. Perplexity explodes without it. learning rates were changed just to emphasize the point. When the learnign rate of 5 is compared to when there was gradient clipping, the model becomes much more unstable much earlier in the training.

ReLU!

We want to see the ReLU function in relation to the tanh function that I implemented previously:

Performance-wise, the two functions appear to perform similarly. However, the relu appears to be a bit more "choppy"

Now, let's compare the training when gradient is not clipped and the learning rate is 1

ReLU seems to handle the data well, even without gradient clipping, though it is visible that the model without gradient clipping is a bit more unstable during its training process, with spikes in the trianing and validation set.

Let's see what happens if we bump up the learning rate:

It is to be expected that training will be more unstable if the learning rate is set to 5, but we can see the effect of the gradient clipping. In the model without clipping, we see a large spike around the 25th epoch. Additional, smaller spikes are seen in later epochs. These perterbations are contianed in this model, but there may be instances in which this is not handled well, and will cause an explosion in perplexity.

Takeaway: We should use gradient clipping in combination with ReLu.

Part 2: GRUs

Task 1: Hyperparameter Tuning

Similarly to part 1, we need to tune some hyperparameters so that we can get some better performance out of our model What we will do differently in this part, is that we will tune one hyperparameter, and then use the best model out of that set to tune the next hyperparameter.

Learning Rate: [.5, 2, 5, 10]

For GRU, there seems to be a much longer, more drawn out tail for the smaller learning rates. In other words, it doesn't want to plateau as quickly as the RNN implementation. As for the best performing model, it seems that a learning rate of 2 provides a good balance of stability and performance, but the performance benefits of the learning rate of 5 outweigh it, so we will go with a LR of 5 for the next trial Let's adjust hiddenUnits now

Hidden Units

Similarly to how the hidden units performed in the RNN implementation, they exhibit a similar pattern here. ore hidden units leads to more complex decision boundaries, and better representation (lower perplexity scores). In these trials, we also see that the learning rate does play a major factor in the stability, as some models (such as the 128 hidden units) is more stable than others (32 hidden units). The model with 64 hidden units performs the best with the current setting of LR = 5 Finally, let's look at minibatch time steps

Minibatch Steps

With minibatch time steps, we see something interesting. The performance of bothe the training and validation set improve as the amount of time steps increases, but after 32 time steps, we see the perplexity of the validation set begin to increase again, while the training set continues to decrease. It seems that the model begins to overfit here, or gets lost with so much context

3 Best Models

All three of these models performed very similarly, with slight deviations between them. The model with the highest amount of hidden units reached the lowest perplexity. The model with 64 hidden units and 16 mb steps performed very comparitively to the previously mentioned model, but had a more closely related validation set The final model has a greater disparity between the perplexity of the training and validation sets.
